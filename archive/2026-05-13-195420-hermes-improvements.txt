❯ Wondering on how to design hermes dashboard. Hermes Dashboard — Design        
  Alternatives                                                                  
  Topology: Three approaches for the initial M-x hermes screen                  
  1. Dedicated dashboard (special-mode), session created on demand              
  - M-x hermes opens *Hermes* dashboard (logo + session list + commands).       
  - User presses RET → session.create → conversation buffer.                    
  - Clean separation: dashboard is a landing page, no session until you ask for 
   one.                                                                         
  - Cons: extra keystroke before chatting, deviates from the official TUI's     
  instant-chat flow.                                                            
  2. Instant session (like the TUI), no dashboard                               
  - M-x hermes immediately spawns gateway + session + *hermes:<sid>* buffer.    
  - What we have right now. Closest to the official React TUI.                  
  - Cons: no session overview, no landing page, no place to show connection     
  status before session is ready.                                               
  3. Hybrid — dashboard with auto-session and embedded composer                 
  - M-x hermes opens *Hermes* dashboard.                                        
  - Gateway starts and session.create fires automatically (like the TUI).       
  - Dashboard shows: logo → connection progression → session info panel (once   
  ready).                                                                       
  - A composer area lives on the dashboard so the user can type immediately.    
  - First prompt auto-creates the real conversation buffer and switches to it.  
  - Dashboard stays open in the background for session overview.                
  ---                                                                           
  Composer UX (for the Hybrid design)                                           
  A. Prompt line in dashboard, input via minibuffer (C-c C-i / i)               
  Dashboard renders "Hermes> _" as the last line of text. Pressing RET or i     
  opens read-string minibuffer. Feels inline because focus transfers            
  immediately. Zero window management. Cons: not truly embedded, one line at a  
  time.                                                                         
  Current code path reused.                                                     
  B. Dedicated composer side-window at the bottom of the frame                  
  ┌─────────────────────┐                                                       
  │  Dashboard  (info)  │                                                       
  ├─────────────────────┤                                                       
  │ Hermes> _           │  ← persistent side window                             
  └─────────────────────┘                                                       
  Closest to the TUI's persistent input. Requires window-split logic, side      
  window config, and focus management. More code but more natural feel.         
  C. Recursive-edit in-buffer composer                                          
  Dashboard buffer has a writable field at the bottom (via field text property  
  + inhibit-read-only). Typing inserts directly into the buffer. Requires       
  careful state management for read-only protection, undo, point motion.        
  ---                                                                           
  Logo / Branding                                                               
  Three-layer priority (like Doom's doom-dashboard-banner-file):                
  1. User Elisp override: (setq hermes-dashboard-logo "...")                    
  2. Skin data from gateway.ready (banner_hero / banner_logo, currently empty   
  in "slate" skin)                                                              
  3. Built-in 6-line "NOUS HERMES" ASCII art from the official TUI's banner.ts  
  ---                                                                           
  Session info panel                                                            
  Once session.info arrives, show below the logo:                               
  - Model name                                                                  
  - Working directory                                                           
  - Session ID                                                                  
  - Tools count / Skills count                                                  
  - System prompt (collapsible)                                                 
  Modeled after the TUI's SessionPanel component (branding.tsx:99-334).         
  ---                                                                           
  Quick actions / Commands section                                              
  Toggleable section at the bottom showing keybindings:                         
  - RET — send prompt / new session                                             
  - i — open minibuffer                                                         
  - s — session list sidebar                                                    
  - g — refresh                                                                 
  - q — quit                                                                    
  - ? — help                                                                    
  Based on the TUI's hotkeys.ts.                                                
  ---                                                                           
  Session switching                                                             
  When sessions exist in hermes--session-buffers, the dashboard lists them with 
   SID / Model / Status / Msgs. RET on a session line switches to that buffer.  
  Pattern from hermes-sessions.el (which would stay as the sidebar              
  alternative).

● For an Emacs client, option 3 (hybrid) with composer UX A (minibuffer) is the
  path of least resistance: you keep the TUI's instant-chat ergonomics (gateway
  + session fire on entry), the dashboard becomes a passive info panel that the
  conversation buffer pops over once the first prompt lands, and you reuse the
  existing hermes-input-send minibuffer flow with zero window-management code.
  Option B (persistent side-window composer) is the prettiest but it's a real
  project — focus stealing, display-buffer-alist tuning, read-only juggling at
  the boundary, and it fights Emacs' "one buffer per task" instinct. Option C
  (in-buffer field) I'd skip; field/inhibit-read-only/undo interactions on a
  live-updating buffer are a known pit.

  Main tradeoff: hybrid+A gives you the dashboard's value (logo, session info,
  connection state, session switcher in one place) without committing to a
  TUI-shaped layout that Emacs users may actively dislike. If later you find the
   minibuffer too transient, you already have hermes-compose (C-c C-l) as the
  multi-line escape hatch — promoting that into a side window is a smaller leap
  than building B from scratch.

  On the logo three-layer priority: worth checking whether banner_hero /
  banner_logo are ever populated in shipped skins before designing around them —
   if they're consistently empty, layer 2 is dead weight and you can collapse to
   user-override → builtin.

  Want me to sketch the dashboard buffer (special-mode, auto session.create,
  logo + info panel + session list + i to prompt), or are you still weighing
  options?

     Plan: doom-dashboard-hermes.el

     Context

     hermes-dashboard.el works in vanilla Emacs but feels sluggish under Doom,
     and the user does not want to risk breaking it while iterating on a
     Doom-styled landing screen. Goal: ship a separate, independent file the
     user can drop into their Doom config (doom-dashboard-hermes.el) that
     renders a Doom-flavored landing buffer for Hermes — centered banner,
     vertical menu, keyboard-driven, with a compact session list.

     The new file MUST NOT touch the existing hermes-dashboard.el, must not
     require it, and must not register hooks that would interfere with the
     vanilla dashboard if both happen to be loaded.

     Why the vanilla dashboard is sluggish (root cause, used to inform the 
     design)

     - hermes-dashboard--refresh does a full erase-buffer + insert + face
     propertization on every trigger.
     - It is wired to hermes-rpc-event-functions AND
     hermes-rpc-connection-functions. Even with the recent event-type filter,
     bursts of session.info/message.complete cause back-to-back full repaints.
     - Layout work (banner split, max-line-width, per-row padding) runs
     unconditionally.

     The Doom version addresses this by debouncing through a 100ms idle timer,
     by rendering to a string first and swapping into the buffer in one
     erase-buffer+insert pair, and by subscribing only to events that change
     visible fields.

     Design

     File: /home/ann/Projects/emacs-hermes/doom-dashboard-hermes.el (new)

     A standalone Doom-styled landing buffer. requires only the data layers:
     hermes-rpc, hermes-state, hermes-input. The conversation-buffer creation
     and primary-session bookkeeping is reused from hermes-mode.el via the
     public hermes, hermes-input-send, hermes-compose, hermes-sessions entry
     points and the hermes--session-buffers map (already used by
     hermes-sessions.el, so the cross-file access is established convention).

     The session list is a single dim line summarising count + gateway state;
     the full table stays in hermes-sessions (l). That keeps the dashboard a
     landing page, not a dashboard-as-monitor.

     Render pipeline (performance-critical)

     1. doom-dashboard-hermes--render builds the full content as a single
     propertized string in a with-temp-buffer, then in the target buffer does
     (let ((inhibit-read-only t)) (erase-buffer) (insert s)) — exactly one
     mutation pair, no per-line insert calls into the live buffer.
     2. Centering uses (/ (- (window-width) line-width) 2) computed once at the
     top of the render, not per-line.
     3. Banner is stripped of Rich […]/[/] markup with one
     replace-regexp-in-string (same approach as the vanilla file's
     --strip-rich). No per-tag face parse — single add-face-text-property over
     the whole banner region.
     4. Cursor is hidden: (setq-local cursor-type nil) + setq-local 
     mode-line-format nil.

     Hook strategy (the real perf fix)

     - One debounced refresh function: doom-dashboard-hermes--schedule-refresh
     cancels any pending run-with-idle-timer and schedules a fresh 0.1s one.
     Bursts coalesce into a single repaint.
     - Subscribe to:
       - hermes-rpc-connection-functions (all states — only fires on
     connect/disconnect, low rate)
       - hermes-rpc-event-functions but filtered to gateway.ready, skin.changed,
      session.info, session.closed, message.complete, error — same allow-list as
      the vanilla file, but enforced before scheduling.
     - window-size-change-functions only re-schedules when the dashboard buffer
     is currently displayed in some window, otherwise no-op.
     - Hooks are added inside doom-dashboard-hermes-mode and removed in a
     kill-buffer-hook so disabling Doom never leaves stale subscribers.

     Primary-session tracking

     The vanilla dashboard's hermes-dashboard--primary-sid is a private
     variable. To avoid coupling, the Doom dashboard derives the "primary"
     session lazily: the most recently active live buffer in
     hermes--session-buffers, picked by sorting on each render. If the user
     wants explicit control later, we expose doom-dashboard-hermes-set-primary —
      out of scope now.

     Result: no need to monkey-patch hermes to record a sid; the dashboard reads
      the map and picks whatever is live.

     Banner

     Default banner = builtin "HERMES" block-letter ASCII (copy of vanilla's),
     since the Rich-marked banner from gateway.ready strips down to nothing
     useful in monochrome. User can override via doom-dashboard-hermes-banner.

❯ Shouldn't we create a state machine with no-sessions/waiting/ready instead?   

● Yes — and ideally at the gateway/transport layer, not per-session. Three
  states:

  - down — process not running
  - starting — process spawned, waiting for gateway.ready
  - ready — accepting requests

  Then hermes-rpc-request becomes the only function that knows about this: if
  ready, send immediately; otherwise enqueue the frame in a pending list and
  flush it when gateway.ready lands. hermes-new-session then doesn't need its
  own one-shot hook — it just calls hermes-rpc-request and trusts the transport.
   commands.catalog, slash.exec, and any future early request get the same fix
  for free.

  Tradeoff: adds one state variable + a small flush function to hermes-rpc.el,
  removes the one-shot trick from hermes-mode.el. Mild abstraction cost, but it
  kills a whole class of races instead of patching one of them. The current
  hermes--last-gateway-ready cache can stay (it's used for skin replay on new
  buffers) but it stops being load-bearing for request scheduling.

  Want me to do it? If so, I'd:
  1. Add hermes-rpc--state (down/starting/ready) and hermes-rpc--pending-frames.
  2. Have hermes-rpc-start set starting, the gateway.ready event set ready and
  flush the queue, and the sentinel reset to down.
  3. Make hermes-rpc-request enqueue when not ready instead of erroring.
  4. Drop the one-shot hook in hermes-new-session.
