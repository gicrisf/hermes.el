  mirrors the official TUI's MessageLine[] + bufRef split, and maps 1:1 to the

  Proposed shape

  (cl-defstruct hermes-state
    connection      ; 'disconnected | 'connecting | 'connected
    session-id      ; string or nil
    session-info    ; plist: model, cwd, tools, skills, lazy
    messages        ; vector of hermes-message — COMMITTED only
    stream          ; hermes-stream or nil — the in-flight turn
    pending         ; hermes-pending or nil — approval/clarify/secret/sudo
    slash-catalog   ; result of commands.catalog
    queue           ; list of queued user inputs
    history         ; ring of past inputs
    skin)           ; plist from gateway.ready

  (cl-defstruct hermes-message
    kind            ; 'user | 'assistant | 'system
    text            ; raw markdown (assistant) / plain (user)
    tools           ; vector of hermes-tool — the "trail" for this turn
    usage           ; plist or nil
    timestamp)

  (cl-defstruct hermes-tool
    id              ; tool_id from gateway
    name            ; "bash", "read", ...
    status          ; 'generating | 'running | 'complete | 'error
    preview         ; latest progress text (during run)
    output          ; final output (on complete)
    error           ; string or nil
    duration        ; seconds (on complete)
    details-mode)   ; 'hidden | 'collapsed | 'expanded

  (cl-defstruct hermes-stream
    text            ; accumulated raw-markdown delta buffer (the "bufRef")
    thinking        ; accumulated thinking.delta
    reasoning       ; accumulated reasoning.delta
    tools)          ; tools attached to THIS in-flight turn

  (cl-defstruct hermes-pending
    kind            ; 'approval | 'clarify | 'secret | 'sudo
    request-id
    payload)

  Why this matches both your goals:

  1. Mirrors the TUI: stream.text is the equivalent of turnController.bufRef. On
   message.complete the reducer commits a fresh hermes-message (with text = 
  stream.text, tools = stream.tools) into messages and sets stream back to nil.
  That's exactly what the official TUI does (turnController.ts:525-537).
  2. Maps to Org without translation: each hermes-message is one top-level
  headline (* user / * assistant); each tools[i] is a nested ** tool: name (Xs)
  subtree; the in-flight stream renders as a sentinel "** [streaming]" headline
  at the end that gets replaced on every delta and removed/promoted on commit.
  No intermediate representation.
  3. Tools-during-streaming live on the stream, not on the previous message —
  mirroring streamingAssistant tools={activeTools}
  (ui-tui/src/streamingAssistant.tsx:50-61).

  The store / dispatch layer

  Buffer-local atom (one per session buffer):

  (defvar-local hermes--state nil)

  (defun hermes-dispatch (msg)
    (let* ((old hermes--state)
           (new (hermes--reduce old msg)))
      (unless (eq old new)
        (setq hermes--state new)
        (run-hook-with-args 'hermes-state-change-hook old new))))

  The process filter routes events to the right buffer by params.session_id,
  then calls hermes-dispatch in that buffer. The renderer is a single hook
  function (hermes--render old new) that diffs and emits Org edits.

  For the verbose copy-* + setf dance, a tiny macro keeps reducer cases
  readable:

  (defmacro hermes--update (state &rest slot-vals)
    "Return a copy of STATE with SLOT-VALS applied."
    ...)

  So message.delta becomes:
  ('message.delta
   (let ((s (or (hermes-state-stream state) (make-hermes-stream :text ""))))
     (hermes--update state
       :stream (hermes--update s :text (concat (hermes-stream-text s) text)))))

● (b) A concrete timeline

  Imagine the user types "list files" and the assistant runs one bash. Here's
  how state evolves under (b1) — tools-on-stream — moment by moment:

  t0  user types "list files"
      → reducer commits user msg:
        messages = [user "list files"]
        stream   = nil

  t1  message.start
        stream = #(stream :text "" :tools [])

  t2  message.delta {text: "Sure, let me "}
        stream.text = "Sure, let me "

  t3  message.delta {text: "check.\n\n"}
        stream.text = "Sure, let me check.\n\n"

  t4  tool.generating {tool_id: "t1", name: "bash"}
        stream.tools = [#(tool :id "t1" :name "bash" :status generating)]

  t5  tool.complete {tool_id: "t1", output: "file1\nfile2"}
        stream.tools[0] now has :status complete :output "..."

  t6  message.delta {text: "Here you go."}
        stream.text = "Sure, let me check.\n\nHere you go."

  t7  message.complete
      → reducer COMMITS the stream:
        messages = [user "list files",
                    assistant "Sure, let me check.\n\nHere you go."
                              tools=[bash(complete)]]
        stream   = nil

  The key property: a tool is never in two places at once. While its turn is
  live, it lives on stream.tools (mutable through copy-and-swap). On
  message.complete, the whole stream — text + tools — freezes into one new
  hermes-message and gets pushed onto messages. After that, the tool entry is
  immutable; it never gets re-rendered unless the user folds/unfolds the subtree
   (which Org handles natively, no state change).

  The renderer benefits: completed messages only need re-rendering when messages
   actually grows (cheap vector-append diff). The in-flight section gets
  rewritten on every delta but only touches the trailing buffer region.

  Contrast with (b2) — a global flat tools vector with parent-pointers: every
  progress event would have to scan/update the right tools entry, and the
  renderer would have to join tools to messages by id on every redraw. Doesn't
  match the Org tree, doesn't simplify anything.

  Does that picture make sense?

● Renderer is the most code-heavy module. Let me sketch the architecture, then
  surface the real decisions.

  Shape

  Two render hooks, one per atom:

  (add-hook 'hermes-state-change-hook    #'hermes--render)
  (add-hook 'hermes-ui-state-change-hook #'hermes--render-ui)

  hermes--render(old, new) is pure dispatch — it inspects which slots changed
  and calls a sub-renderer per slot:

  (defun hermes--render (old new)
    (with-silent-modifications
      (save-excursion
        ;; messages grew → append the new tail message(s)
        (when (> (length (hermes-state-messages new))
                 (length (hermes-state-messages old)))
          (hermes--render-new-messages old new))
        ;; stream lifecycle
        (let ((os (hermes-state-stream old))
              (ns (hermes-state-stream new)))
          (cond ((and (null os) ns)   (hermes--stream-begin ns))
                ((and os (null ns))   (hermes--stream-commit))   ; just drop
  markers
                ((not (eq os ns))     (hermes--stream-update os ns))))
        ;; pending prompt → trigger minibuffer (in a timer, not here)
        (unless (eq (hermes-state-pending old) (hermes-state-pending new))
          (hermes--maybe-trigger-pending new))
        ;; session-info → header-line
        (unless (eq (hermes-state-session-info old) (hermes-state-session-info
  new))
          (hermes--render-header new)))))

  Each sub-renderer touches only its region, located via persistent markers (not
   text-property scanning).

  Streaming: stable/unstable split, mirroring the TUI

  The official TUI's streamingMarkdown.tsx:107 splits at the last \n\n that's
  outside a fenced code or math block. We mirror that. Two buffer-local markers
  bound the in-flight region:

  (defvar-local hermes--stream-stable-end nil)  ; everything before this is
  frozen
  (defvar-local hermes--stream-end nil)         ; end of unstable tail

  On every message.delta:
  1. Compute new stable boundary in stream.text (last \n\n outside ``` fences).
  2. If it advanced: insert the newly-stable substring at --stream-stable-end,
  advance the marker.
  3. Replace the region [--stream-stable-end, --stream-end) with the new
  unstable suffix.

  On message.complete: drop the markers. The streaming region already has the
  final format, no rewrite needed. Streaming and committed rendering use the 
  same format on purpose — commit becomes a no-op buffer-wise.

  The decisions I want your call on

  (R1) Tool placement: trail vs interleaved

  The original draft (emacs-hermes.md:127-143) showed tools interleaved with
  assistant text — paragraph, bash subtree, paragraph, read subtree. But the
  official TUI doesn't do that: tools are a "trail" — a single kind: 'trail'
  system message rendered after the assistant text (messageLine.tsx:67-90). They
   appear as a contiguous block of tool entries below the assistant prose.

  State-shape-wise: hermes-message.tools is a flat vector with no positional
  info between tools and text. So we literally cannot interleave without
  changing the state model to something like :body [(text "...") (tool ...) 
  (text "...")].

  I'd go trail (TUI-faithful, simpler state, matches tool.start not carrying any
   position info from the gateway anyway). Your draft says interleave — was that
   intentional, or just an artifact of sketching?

  (R2) Read-only conversation history?

  The Org buffer renders the conversation. Two stances:
  - Read-only history + editable input area at bottom (comint/SLIME style): user
   types at the end, can't accidentally edit past messages, can still copy/yank.
  - Fully editable Org buffer: user can mess with anything, including past
  messages. Bad for a live conversation but very Emacs-y.

  I'd lean read-only history with a separate input area. But this couples
  renderer to the input layer, so we should decide together.

  (R3) Markdown rendering inside Org

  We accumulate raw markdown into the buffer (decided earlier). Three
  sub-options for how it lives in Org:
  - Plain text (markdown chars visible as-is — **bold** shows as **bold**)
  - Convert to Org syntax on commit (markdown→org converter; cheap for inline,
  hard for tables/code)
  - Wrap in a markdown-mode overlay via polymode or a minor mode that fontifies
  markdown inside an Org region

  The TUI tokenizes markdown for ANSI display — that doesn't translate. Plain 
  text for v1 is honest and zero-effort; we can layer fontification later.
  Agree?

  (R4) Point preservation during streaming

  If the user is scrolled up reading earlier messages while a turn streams
  below, we must not move point. save-excursion handles position; but the buffer
   grows, so the window-start may drift. Standard fix: only auto-follow the tail
   if point is already at point-max (the "tail-mode" trick from
  comint-scroll-to-bottom-on-output). Sound right?

  (R5) Markers vs text properties for locating regions

  We need to find: "the tool subtree for tool_id=t1", "the assistant headline at
   index N", "the streaming region". Two options:
  - Markers: per-tool, per-message; cheap to update, but proliferate (~N markers
   per session).
  - Text properties + scanning: text-property-any + org-find-property; no
  bookkeeping, but O(N) per lookup.

  Default: markers for streaming region, text-properties (:tool_id Org property)
   for committed tools. Lookups for committed tools are rare (only when status
  updates arrive after commit — which the TUI doesn't handle anyway, per
  edge-case #2). Agree?

● Renderer locked. Last big topic before exit: input layer. Three pieces tangle
  together:

  (I1) Where does the user type?

  Now that the conversation buffer is read-only, the composer needs a home.
  Three patterns:

  - (a) Comint-style bottom prompt — single buffer, read-only history above,
  editable input region after a Hermes> prompt at end. Submits on RET, M-RET for
   newline. SLIME-style. Tightest UX, most code.
  - (b) Separate composer buffer below — *hermes-input* window in a side window

  - (c) Minibuffer prompt — read-string each time. Lispy, zero buffer overhead,
  but can't compose multi-line, can't show queue.

  I'd lean (b) for v1 — clean separation, decent UX, much less code than (a).
  (a) is the long-term target.

  (I2) Queue semantics

  When the user submits while a turn is live (stream non-nil), the input gets
  queued. Drain on message.complete: pop the head and dispatch as prompt.submit.
   Slash commands bypass the queue entirely (commands.catalog lookup →
  slash.exec immediately, even mid-turn).

  The queue lives on hermes-state.queue (already in the shape). Visible? I'd
  render it as a footer/header-line indicator: Queue: 2 messages rather than
  inline in the buffer. Sound right?

  (I3) Slash commands

  Tab-completion against the cached commands.catalog. The catalog is fetched
  once on connect and cached on hermes-state.slash-catalog. The composer detects
   a leading /, switches completion-at-point-functions to a hermes-specific one.

     Corrections to the draft

     Event/method names use dots, not dashes. Every outgoing method in the
     draft is wrong: user-input → prompt.submit, approval-respond →
     approval.respond, clarify-respond → clarify.respond, secret-respond →
     secret.respond, sudo-respond → sudo.respond, slash-exec →
     slash.exec, session-create → session.create, session-resume →
     session.resume.

     Event envelope is wrapped, not bare. Server emits
     {"jsonrpc":"2.0","method":"event","params":{"type":"<name>","session_id":"<
     sid>","payload":{...}}}
     (tui_gateway/server.py:382-386). The reducer should dispatch on
     params.type, not on a top-level type field.

     tool.start does not exist. The real event is tool.generating
     (server.py:1599). Other tool events are tool.progress
     ({name, preview}) and tool.complete ({name, output, error?, exit_code?}).

     Missing events the draft does not list but must handle:
     error ({message}, server.py:3012), skin.changed, reasoning.available,
     browser.progress, voice.status, voice.transcript. Voice/browser can be
     no-ops for v1.

     gateway.stderr and gateway.protocol_error are NOT gateway events —
     they are TUI-side conventions. Emacs can synthesize them similarly by
     reading
     the subprocess stderr and catching JSON parse errors in the process filter.

     Missing outgoing RPCs: session.close, session.interrupt (for Ctrl-C —
     answers draft open-question #8), command.dispatch, commands.catalog
     (returns {pairs, sub, canon, categories, skill_count, warning} —
     server.py:4272-4368; this is how the draft's slash-commands cache is
     populated).

     approval.respond payload is {session_id, choice, all?} not boolean
     allow. choice carries allow/deny/all semantics.

     clarify.respond uses answer, not choice index.

     Markdown rendering: gateway sends raw markdown in text, plus an
     optional rendered (legacy Rich ANSI — ignore it). The Emacs client should
     accumulate the raw markdown text
     (ui-tui/src/turnController.ts:525,533-537).
     This resolves draft open-question #3: it's not HTML; do not need an
     #+begin_export block.

     Request/response IDs and async dispatch: every request carries id;
     responses echo it (server.py:427). Eight long-running methods
     (cli.exec, slash.exec, shell.exec, session.resume, session.branch,
     session.compress, skills.manage, browser.manage) are dispatched to a
     thread pool — the worker writes the response asynchronously
     (server.py:146-156). Emacs RPC layer must support pending request map keyed
     by id rather than expecting synchronous request/response interleaving.

     Blocking prompts (clarify.request, sudo.request, secret.request)
     block a gateway thread on threading.Event until the TUI sends the matching
     .respond with the same request_id (server.py:723). The draft is right
     to keep request_id in the pending state.

     Session running flag: session["running"] prevents concurrent
     prompt.submit (server.py:3003-3004). The Emacs queue logic must mirror
     this — never send a second prompt.submit while the previous turn is live.

     State architecture (locked)

     Two atoms per session buffer — a clean split between persistent state
     (diffed into the Org buffer) and ephemeral UI state (drives mode-line /
     overlays / spinners, never touches Org). Both are cl-defstructs mutated
     through copy-and-swap; renderers subscribe via hooks.

     (cl-defstruct hermes-state            ; persistent — mirrored in Org
       connection                          ; 'disconnected | 'connecting |
     'connected
       session-id session-info             ; session.info payload
       messages                            ; vector of hermes-message (COMMITTED
      only)
       stream                              ; hermes-stream or nil (in-flight
     turn)
       pending                             ; hermes-pending or nil
       slash-catalog queue history skin)

     (cl-defstruct hermes-message          ; one Org top-level headline
       kind                                ; 'user | 'assistant | 'system
       text                                ; raw markdown
       tools                               ; vector of hermes-tool — the trail
       usage timestamp)

     (cl-defstruct hermes-tool             ; one Org subtree under its message
       id name
       status                              ; 'generating | 'running | 'complete
     | 'error
       output error duration)
     ;; Note: NO preview, NO details-mode. Preview is ephemeral (ui-state).
     ;; Fold/unfold is org-cycle's job, buffer-local, not in our state.

     (cl-defstruct hermes-stream           ; the in-flight turn — TUI's "bufRef"
       text thinking reasoning             ; accumulated deltas
       tools)                              ; tools attached to this live turn

     (cl-defstruct hermes-pending
       kind                                ; 'approval | 'clarify | 'secret |
     'sudo
       request-id payload)

     (cl-defstruct hermes-ui-state         ; ephemeral — never persisted
       status-text status-kind             ; from status.update
       spinner-frame                       ; advanced by timer
       tool-previews)                      ; alist tool-id → latest preview

     Buffer-local atoms + zustand-style dispatcher:

     (defvar-local hermes--state nil)
     (defvar-local hermes--ui-state nil)

     (defun hermes-dispatch (msg)
       (let* ((old hermes--state)
              (new (hermes--reduce old msg)))
         (unless (eq old new)
           (setq hermes--state new)
           (run-hook-with-args 'hermes-state-change-hook old new))))

     (defun hermes-ui-dispatch (msg)
       (let* ((old hermes--ui-state)
              (new (hermes--ui-reduce old msg)))
         (unless (eq old new)
           (setq hermes--ui-state new)
           (run-hook-with-args 'hermes-ui-state-change-hook old new))))

     Renderer (locked)

     Two render hooks, one per atom:

     (add-hook 'hermes-state-change-hook    #'hermes--render)
     (add-hook 'hermes-ui-state-change-hook #'hermes--render-ui)

     hermes--render(old, new) is pure dispatch — inspects which slots changed
     and calls a sub-renderer per slot. Each sub-renderer touches only its
     region.
     Wrap edits in with-silent-modifications + save-excursion to avoid marking
     the buffer dirty and to preserve point.

     (defun hermes--render (old new)
       (with-silent-modifications
         (save-excursion
           (when (> (length (hermes-state-messages new))
                    (length (hermes-state-messages old)))
             (hermes--render-new-messages old new))
           (let ((os (hermes-state-stream old))
                 (ns (hermes-state-stream new)))
             (cond ((and (null os) ns) (hermes--stream-begin ns))
                   ((and os (null ns)) (hermes--stream-commit))
                   ((not (eq os ns))   (hermes--stream-update os ns))))
           (unless (eq (hermes-state-pending old) (hermes-state-pending new))
             (hermes--maybe-trigger-pending new))
           (unless (eq (hermes-state-session-info old)
                       (hermes-state-session-info new))
             (hermes--render-header new)))))

     "bar"

     "baz"

     indicator when empty.

     slash commands dispatch immediately. The gateway's session-running flag

     handler dispatched async (server.py:146), so it doesn't compete with the
     streaming turn.
